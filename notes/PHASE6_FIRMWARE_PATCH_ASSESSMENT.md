## 2. 방법 A: DFU 펌웨어 교체 (가장 현실적)

### 2.1 STM32H745 부트 메커니즘

```
Power On
│
├── BOOT0=0, BOOT1=0 → Flash Bank 1 (CM7: 0x08000000)
│                     → Flash Bank 2 (CM4: 0x08120000)
│
├── BOOT0=1, BOOT1=0 → System Bootloader (USART/USB DFU)
│
└── DFU Mode (IF#1 WINUSB)
    └── TUSBAUDIO API로 펌웨어 플래싱
```

### 2.2 펌웨어 업데이트 경로 (Phase 5에서 확인)

DLL에서 3가지 경로를 확인:
1. **DFUUpdater** — TUSBAUDIO.dll 사용, IF#1 WINUSB
2. **RKUpdater** — Rockchip 시리얼 부트로더
3. **CollageUpdater** — USB Bulk (Protobuf)

### 2.3 펌웨어 교체 시나리오

```
현재 펌웨어 (4.0.1):
├── CM4: 608KB @ 0x08120000 (Flash Bank 2)
├── CM7: 512KB @ 0x08000000 (Flash Bank 1)
├── FX:   120KB (별도 코어)
└── UI:   4개 MCU (각각 독립)

커스텀 펌웨어 제작 워크플로우:
1. 기존 펌웨어 덤프 (DFU Upload)
2. Ghidra에서 분석 (Phase 1-5 완료)
3. 수정 포인트 식별 (하단 3.3 참조)
4. 패치 적용 (바이너리 에디팅 또는 재컴파일)
5. CRC/Fingerprint 재계산 (필요시)
6. DFU Download으로 플래싱
7. 검증 및 테스트
```

### 2.4 복구 가능성

| 시나리오 | 복구 방법 |
|----------|-----------|
| 펌웨어 오류로 부팅 실패 | DFU 모드 진입 (BOOT0 핀) → 공식 펌웨어 재플래시 |
| 브릭 (완전 고장) | ST-Link JTAG/SWD로 직접 플래싱 |
| 보증 무효화 | Arturia 서비스 센터 방문 필요 |

**★ STM32H7은 항상 System Bootloader로 부팅 가능 (BOOT0=1). 완전한 브릭은 거의 불가능.**

## 3. 수정 가능한 대상 분석

### 3.1 CM7 DSP 코어 — 오디오 엔진

| 수정 대상 | 가능성 | 난이도 | 설명 |
|-----------|--------|--------|------|
| 오실레이터 파라미터 | ✅ | ★★ | Init 함수 테이블 수정으로 파라미터 범위/기본값 변경 |
| 필터 타입 추가 | ⚠️ | ★★★ | 기존 vtable에 새 엔진 추가. Plaits 아키텍처 제약 |
| LFO 쉐이퍼 수정 | ✅ | ★★ | Shp1_*/Shp2_* 테이블 데이터 교체 |
| 보이스 수 변경 | ⚠️ | ★★★ | 12보이스 하드코딩 가능성. SRAM/CPU 부하 영향 |
| FX 라우팅 변경 | ❌ | ★★★★ | FX 코어가 분리됨. 프로토콜 미상 |
| 새로운 DSP 알고리즘 | ⚠️ | ★★★★ | 빈 함수 슬롯이나 hook point 필요 |

**CM7 핵심 함수 (Phase 4에서 식별):**
```
FUN_080359f4 (18KB) — Main Render Loop → 수정 시 전체 사운드 변화
FUN_0803e6f8 (10KB) — Osc Mix
FUN_080321d4 (8KB)  — Envelope
FUN_0803c2bc (9KB)  — VCF Filter
FUN_0803a490 (8KB)  — VCA
FUN_08054708 (7KB)  — Mod Matrix
```

### 3.2 CM4 I/O 코어 — 페리페럴 관리

| 수정 대상 | 가능성 | 난이도 | 설명 |
|-----------|--------|--------|------|
| MIDI CC 매핑 변경 | ✅ | ★★ | FUN_08166810의 switch/case 테이블 수정 |
| SysEx 핸들러 추가 | ✅ | ★★ | FSM (43-state)에 새 state 추가 |
| USB 디스크립터 변경 | ⚠️ | ★★★ | VID/PID 변경 시 Windows 드라이버 이슈 |
| SAI 클럭 변경 | ⚠️ | ★★★ | 샘플레이트 변경. DAC/ADC 호환성 영향 |
| MIDI 라우팅 모드 | ✅ | ★★ | "USB Only"/"MIDI DIN Only" 문자열 참조 |
| 새로운 USB 엔드포인트 | ❌ | ★★★★ | USB 스택 전체 수정 필요 |

### 3.3 가장 현실적인 수정 (Low-Hanging Fruit)

#### 1. MIDI CC 리매핑 (가장 쉬움)
```
대상: FUN_08166810 (CC handler, CM4)
방법: switch/case 테이블에서 CC# → 파라미터 매핑 변경
예: CC#24 (Cutoff) → 다른 파라미터로 변경
복잡도: 바이너리 패치 몇 바이트
위험도: 낮음 (MIDI만 영향)
```

#### 2. 오실레이터 기본 파라미터 변경
```
대상: 13개 VTable Init 함수 (0x37CB4~0x386D4, CM7)
방법: Group A/B/C Init 함수의 파라미터 테이블 수정
예: SuperWave의 unison 수 기본값 변경
복잡도: Ghidra에서 데이터 섹션 수정
위험도: 낮~중간 (DSP 수치 이슈 가능)
```

#### 3. LFO 쉐이퍼 곡선 변경
```
대상: Shp1_*/Shp2_* 파라미터 (130개)
방법: .mnfx 또는 펌웨어 내부 파라미터 테이블 수정
복잡도: 파라미터 값만 변경하면 됨
위험도: 매우 낮음
```

#### 4. 프리셋 확장
```
대상: 512 프리셋 슬롯 (0xD00 바이트 each)
방법: 펌웨어의 프리셋 슬롯 수 상수 변경
예: 512 → 1024 슬롯 (플래시 여유공간 확인 필요)
복잡도: 상수 변경 + 프리셋 관리 로직 수정
위험도: 중간 (플래시 용량 초과 시 브릭)
```

## 4. 기술 장벽

### 4.1 해결 가능한 장벽

| 장벽 | 해결 방법 | 노력 |
|------|-----------|------|
| CRC/서명 검증 | 펌웨어 덤프 후 CRC 알고리즘 역추적 | ★★ |
| 펌웨어 암호화 | Phase 5에서 CRYPT32.dll 참고. 암호화 위치 확인 필요 | ★★★ |
| 부트로더 잠금 | BOOT0 핀으로 우회 가능 | ★ |
| 컴파일 환경 없음 | 바이너리 패치로 대체. 재컴파일은 ARM GCC + STM32H7 HAL로 가능 | ★★★ |

### 4.2 해결 어려운 장벽

| 장벽 | 이유 | 대안 |
|------|------|------|
| DAC/ADC 칩 미식별 | 펌웨어에 칩 이름 없음 | 물리 보드 분석 필요 |
| CM4↔CM7 인터코어 프로토콜 | 간접 포인터 참조, Ghidra에서 추적 불가 | JTAG 런타임 트레이스 |
| FX 코어 인터페이스 | DSP56362, 프로토콜 미상 | FX 코어 펌웨어 역공학 필요 |
| UI MCU 통신 | 4개 독립 MCU, 프로토콜 미상 | I2C/SPI 로직 분석기 필요 |
| Collage Protobuf .proto | DLL 바이너리에서 field numbers 추출 필요 | protobuf-decoder 사용 |
| ASC 보호 | Arturia Software Center 연동 | 패치 우회 또는 비활성화 |

## 5. 위험 평가

### 5.1 브릭 위험 매트릭스

```
              수정 복잡도
              Low     Med     High
           ┌──────┬──────┬──────┐
    Low    │  🟢  │  🟢  │  🟡  │  복구 가능성
    Risk   │ CC   │ Init │ SRAM │
           │ remap│ tbl  │ mod  │
           ├──────┼──────┼──────┤
    Med    │  🟡  │  🟡  │  🟠  │
    Risk   │ LFO  │ Voice│ SAI  │
           │ shp  │ alloc│ clock│
           ├──────┼──────┼──────┤
    High   │  🟠  │  🔴  │  🔴  │
    Risk   │ Preset│ Boot │ USB  │
           │ slot │ code │ desc │
           └──────┴──────┴──────┘

🟢 = DFU로 복구 가능
🟡 = ST-Link으로 복구 가능
🟠 = 하드웨어 수정 필요 가능성
🔴 = 영구 브릭 위험 (낮음)
```

### 5.2 보증 영향

- **DFU 모드 진입**: 물리적 조작 필요 → 보증 흔적 남을 수 있음
- **펌웨어 해시 불일치**: ASC에서 감지 가능
- **공식 펌웨어 업데이트**: 커스텀 펌웨어로 덮어씌워짐

## 6. 권장 접근 전략

### Phase A: 안전한 바이너리 패치 (1-2주)

```
1. 펌웨어 DFU 덤프 (기존 펌웨어 백업)

   - 오실레이터 파라미터 기본값
   - LFO 쉐이퍼 데이터

5. DFU 플래싱
6. 테스트
```

## 8. 결론

**MiniFreak 펌웨어 수정은 기술적으로 가능하며, STM32H7의 표준 부트로더와 DFU 지원 덕분에 브릭 위험이 관리 가능한 수준입니다.**

가장 현실적인 첫 단계:
1. **DFU 덤프**로 펌웨어 백업
2. **MIDI CC 리매핑**으로 간단한 바이너리 패치 시도
3. 성공 시 **오실레이터 파라미터** 및 **LFO 쉐이퍼** 수정으로 확장

주요 리스크는 DAC/ADC 칩 미식별과 FX 코어 프로토콜 미상이며, 이는 물리적 보드 분석으로 해결해야 합니다.
